         def read_pipe_delimited(filepath, fieldnames):
             """
             Reads a pipe delimited file into a list of dicts keyed by fieldnames
             """
             data = []
             if not os.path.exists(filepath):
                 return data
             with open(filepath, 'r', encoding='utf-8') as f:
                 for line in f:
                     line = line.strip('\n').rstrip('|').strip()
                     if not line:
                         continue
                     parts = line.split('|')
                     if len(parts) < len(fieldnames):
                         # Fill missing parts with empty strings
                         parts += [''] * (len(fieldnames) - len(parts))
                     entry = dict(zip(fieldnames, parts))
                     data.append(entry)
             return data

             if count > 0:
                 avg_time = round(total_avg_time / count, 2)
             else:
                 avg_time = 0.0
         
             analytics_summary = {
                 'total_views': total_views,
                 'total_unique_visitors': total_unique_visitors,
                 'average_time_on_article': avg_time,

         def read_pipe_delimited(filepath, fieldnames):
             """
             Reads a pipe delimited file into a list of dicts keyed by fieldnames
             """
             data = []
             if not os.path.exists(filepath):
                 return data
             with open(filepath, 'r', encoding='utf-8') as f:

                     parts = line.split('|')
                     if len(parts) < len(fieldnames):
                         # Fill missing parts with empty strings
                         parts += [''] * (len(fieldnames) - len(parts))
                     entry = dict(zip(fieldnames, parts))
                     data.append(entry)
             return data

                     if ap['article_id'] == article_id and ap['version_id'] == version_id:
                         if ap['approver'] == stage and ap['status'].lower() == 'approved':
                             approved = True
                             break
                 if not approved:

         def update_article_status_based_on_approval(article, category, article_id, version_id):
             # If fully approved, mark article status to 'published', else 'in_review' if any approval exists else 'draft'
             fully_approved = is_version_fully_approved(category, article_id, version_id)
             if fully_approved:
                 article['status'] = 'published'
             else:
                 # Check if any approval record exists for this article/version
                 approvals = load_approvals()
                 any_approval = any((ap['article_id'] == article_id and ap['version_id'] == version_id) for ap in approvals)
                 if any_approval:
                     article['status'] = 'in_review'
                 else:
                     article['status'] = 'draft'

             new_version_id = generate_new_version_id(article_versions)

             if count > 0:
                 avg_time = round(total_avg_time / count, 2)
             else:

             analytics_summary = {
                 'total_views': total_views,
                 'total_unique_visitors': total_unique_visitors,
                 'average_time_on_article': avg_time,

         Functional Testing Feedback for ContentPublishingHub
         
         1. Dashboard Page (/dashboard):
         - The route returns 200 OK and includes the username and quick stats.
         - Quick Stats correctly counts total articles and categorizes them by status.
         - Recent activity lists the last 5 article versions.
         
         2. Create Article (/article/create):
         - GET shows the create article form.
         - POST with missing title, content, or category correctly returns errors.
         - POST with valid data creates article and initial version, returns success message.
         
         3. Edit Article (/article/<id>/edit):
         - GET on valid article returns latest version content and version number.
         - GET on nonexistent article returns 404.
         - POST with empty content returns validation error.
         - POST with content updates article by adding new version and sets status to draft.
         
         4. Version History (/article/<id>/versions):
         - GET on valid article returns sorted version list with approvals and comments count.
         - GET on nonexistent article returns 404.
         
         5. My Articles and Published Articles (/articles/mine, /articles/published):
         - Filtering by status on my articles works.
         - Filtering and sorting by category or date/title works on published articles.
         
         6. Content Calendar (/calendar):
         - GET shows calendar data grouped by scheduled publish dates.
         - POST scheduling updates article last_modified_date if inputs valid.
         - POST with missing or invalid inputs returns errors.
         
         7. Article Analytics (/article/<id>/analytics):
         - Shows aggregation of analytics data including total views, unique visitors, average time, and shares.
         - 404 on invalid article ID.
         
         Defects and Issues:
         1. In app.py, in /calendar route the form input names in HTML (schedule-article-id, schedule-date) do not match those expected on POST ('article_id', 'new_date').
            This causes scheduling updates to fail silently or show error 'Article and date are required for scheduling.'
            Suggest aligning form input names with backend or adjusting backend to match frontend.
         
         2. Templates have mismatches in variable names and usage:
         - dashboard.html recent_activity list renders each activity as {{ activity }}, but activity is dict, should access expected keys.
         - edit_article.html form action uses article['id'] but article dict has 'article_id' as key.
         - similarly other templates use inconsistent keys (id vs article_id, last_modified vs last_modified_date).
         - article_analytics.html uses article_id undefined in URLs, should use article.article_id.
         
         3. Some templates have missing element IDs that are required by spec:
         - dashboard.html lists quick stats and recent activity but they do not have element IDs for each data item.
         - content_calendar.html uses 'status-msg' and 'validation-errors' but backend sends 'status_message' and 'validation_errors' (name mismatch).
         
         4. The /article/<id>/edit POST does not update workflow or approvals, may cause inconsistent status states.
         
         5. Approvals and comments management routes and UI elements appear missing for approval flow testing.
         
         Summary:
         The core backend routes and logic mostly function correctly for article creation, editing, version history, and basic filtering.
         However, there are inconsistencies and bugs in form input naming, template variable references, and missing element IDs which affect UI behavior and backend integration.
         Corrections should focus on syncing form names and backend reading, fixing template variable keys, and ensuring all UI elements needed for interaction have correct IDs.
